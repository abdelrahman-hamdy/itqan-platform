E. Critical caveat: audit --apply cannot tell admin-preset from phantom

The audit command does not know about admin presets. It computes expected = COUNT(platform sessions completed + counted) and would force cycle.sessions_used to that value. For a cycle that legitimately holds 4 platform + 8 admin-preset = 12, applying the audit would overwrite the 12 with 4 — destroying the admin-preset history that represents real money paid for pre-platform tutoring.

This is why I'm not recommending a blanket audit-cycle-counts --apply run, even per-subscription. The right approach is:

  1. Najah-style cases (positive drift) are safe — apply via the audit command per subscription. The audit only ever raises the counter to match completed sessions, never lowers below admin presets.
  2. Maryam-style cases (negative drift on active cycle) need manual SQL. Decide which cycle has the phantom and update cycle.sessions_used + cycle.sessions_completed directly. If it's the active cycle, also mirror onto subscription's sessions_used / sessions_remaining.
  3. Archived cycles with negative drift — leave alone unless you can confirm no admin preset existed for that cycle.
